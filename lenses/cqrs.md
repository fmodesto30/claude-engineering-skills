# CQRS Lens

Reusable Command Query Responsibility Segregation knowledge for Java/Spring: when separating the model that handles writes from the model that serves reads pays for itself, and whether the full separate-store form is warranted at all. **The intent of using this lens comes from the consuming skill, not from the lens itself.**

## How skills use this lens

Today this lens has one consumer: **`architecture-review`**, which applies it at **system/design altitude** — whether the read and write paths of a piece of the system genuinely diverge enough to model separately, where the read model is derived from and how it stays in sync, what staleness the read side exposes, and whether a separate read store earns the eventual consistency it costs — not line-by-line PR nitpicking. The review-flavored sections below (severity calibration, comment style, integration contract) are written for that design-review intent: *do the reads and writes here diverge enough in shape, scale, or query complexity that separating them buys something concrete, or is the design paying for a projection, a sync mechanism, and eventual consistency that a single model against one store would make unnecessary?*

The same knowledge could later serve a smaller-altitude diff reviewer — a single PR that adds a Spring Data projection, a read-only query repository, or a materialized view — with a *different*, narrower intent. That is why it lives in `lenses/`. But the lens is **not** generalized for that second consumer now: it stays focused on the design-review questions a reviewer can answer from a read/write workload picture and a sync model, and a neutral core is extracted only when a second real consumer exists. A consuming skill loads it only when the design or change actually involves a divergence between read and write models as described in "When to Use" below — never just because the words "CQRS," "read model," or "projection" appear in a name, a package, or a design doc.

## Purpose

This lens helps the reviewer judge whether separating a write model from a read model actually serves the system, and whether the heavyweight separate-store form is the right instrument for that separation at all. Its job is to separate a genuine read/write divergence — a write path enforcing invariants whose shape is wrong for the query the UI needs, a read workload that dwarfs the write workload, a report that cannot be served from the aggregate shape without an N+1 storm — from CQRS ceremony bolted onto ordinary CRUD whose reads and writes are the same shape and the same scale. The default bias is strongly toward the simplest model that satisfies the requirement: **for symmetric CRUD, one model serving both reads and writes against one store is correct, and CQRS — especially a separate read store — is overengineering.** When the read and write needs genuinely diverge, the lens gives the reviewer a consistent way to ask "what does separating them buy, what does the read model derive from, how does it stay in sync, and what staleness does that introduce?" and to phrase the answer as a consequence-anchored finding rather than a pattern preference. It is equally a tool for restraint — for confirming that a single model is correct and that the right recommendation is *against* CQRS, or at most toward a lightweight read-only projection in the same store.

## When to Use

Consult this lens when the design or change involves a genuine divergence between how data is written and how it is read, including:

- The **read and write workloads diverge sharply** in shape, scale, or query complexity — reads vastly outnumber writes (or vice versa), or the queries need a shape the write model does not have.
- A **denormalized or derived read model** is introduced — a precomputed, flattened, or aggregated view shaped for a specific consumer rather than for enforcing invariants.
- A **separate read store or read replica** is added to serve queries away from the write store.
- **Multiple distinct read views** of the same underlying data are needed (a customer-facing `Order` summary, an operational dashboard, a `Report`), each wanting a different shape.
- A **complex query for reporting or display** is being forced through the aggregate/write model and paying for it (cross-aggregate joins, N+1 loads, loading whole aggregates to read one field).

Do NOT engage this lens merely because the vocabulary appears. A class named `OrderReadModel`, a package called `query`, a Spring Data projection interface, or a `@Query` method is not itself a reason to engage — the trigger is a real divergence between the write path and the read path that the design is responding to, not the words used to label it. **And do NOT engage merely because** the system uses DDD or is split into microservices: aggregates are the write model and a service boundary is not a read/write split. Neither implies CQRS, and adopting CQRS "because we already do DDD/microservices" is the pattern-by-association mistake this lens exists to catch.

## Core Principle

CQRS is a tool, not an objective. It separates the model that handles **commands** — writes and state changes, the path that enforces invariants and decides what is allowed — from the model that serves **queries** — reads, shaped for how a consumer wants to view the data. The fact that a design matches the CQRS shape is never, by itself, a reason to adopt it.

Crucially, **CQRS is a spectrum, not a binary**:

- The **lightweight** form is just separate read and write methods, or read-only query models, against the **same datastore**. The command side loads and saves aggregates through their repositories; the query side reads through dedicated read-only projections or query methods that bypass the aggregate. There is one source of truth and one store: a non-materialized projection or plain view is recomputed from the write data on each query, so it reads as fresh as the writes with no eventual consistency. (A *materialized* same-store view is still derived, so it is stale between refreshes — a bounded, refresh-driven staleness, distinct from the unbounded asynchronous lag of a separate store.)
- The **heavyweight** form is a **separate, denormalized read store** kept in sync with the write store **asynchronously**. The command side writes to the write store; a projection process derives the read model and updates the read store after the fact. This buys read scalability (the read store can be scaled, indexed, and shaped independently) and query-shape freedom (the read model can be denormalized for exactly the queries it serves) — at the cost of **eventual consistency** (reads lag writes), **a projection to build and maintain**, and **two models to keep in sync**.

A separation earns its keep only when it does concrete work the single model cannot: the query shape the consumers need is genuinely wrong for an invariant-enforcing write model and contorts it; the read workload must scale independently of the write workload; or several distinct read views of the same data each want their own shape. The heavyweight separate-store form is justified only when those needs are real *and* the system can tolerate the read side being eventually consistent — and that bargain must be made consciously.

**For ordinary CRUD with symmetric read/write needs, a single model is correct and CQRS is overengineering.** This is the most common correct finding for this lens. A single model against one store reads-its-own-writes for free, has no projection to maintain, no sync mechanism to operate, and one place where the data lives; spending eventual consistency, a projection, a sync pipeline, and two models on a screen that just shows back the fields it just saved is pure cost. This lens leans hard against dogma: CQRS — and the separate-store form most of all — is among the most over-applied patterns in this set. **"No comment" — and "this doesn't need CQRS" — are valid, frequent, and correct outcomes**, and recommending a lightweight read-only projection in the same store *instead* of a separate read store is itself a finding, not silence.

## Severity Calibration

Apply these four levels to CQRS findings (see [`../rules/severity-rubric.md`](../rules/severity-rubric.md) for the shared definitions):

- **MUST** — A separation decision causes a concrete defect or serious hazard now: a separate read store updated asynchronously where the UI reads it back **immediately after the command** and treats a stale read as the truth, so the user sees their just-made change missing (a read-your-own-writes anomaly that looks exactly like a lost write — wrong decisions, support tickets); a read projection with **no clear owner and no rebuild path** that has silently drifted from the write model, so the system serves wrong data with no way to diagnose or correct it; a projection updated by a **dual write** in the same code path as the command with no atomic guarantee, so a crash leaves the read model permanently inconsistent with the write model; the command side reading through the **read model to make a decision and enforce an invariant against a stale view**, so two concurrent commands both pass a check the fresh state would have failed; complex multi-aggregate reporting forced through the **write/aggregate model** producing an N+1 or whole-aggregate-load storm that takes a critical query down under load. These must be raised and must explain the failure mode.
- **SHOULD** — No defect today, but a specific change would clearly improve correctness-under-staleness, scalability, or comprehension, and you can state the concrete benefit: a heavyweight read store introduced where a **read-only projection in the same store** would serve the same query without eventual consistency and without a sync pipeline to operate; a reporting query loading whole aggregates to read a few fields where a **dedicated read-only projection** would avoid the over-fetch before the dataset grows; a projection whose **rebuild is possible but undocumented and untested**, so a future drift would be recoverable only by guesswork; a staleness window that is currently tolerable but unbounded and unmonitored. Raise it when the benefit is real and nameable.
- **NIT** — Naming of a read model or projection, the package a query model lives in, or an optional structural preference with no behavioral consequence (calling a type `OrderSummaryView` vs `OrderReadModel`). Worth a brief note, never a blocker.
- **NO_COMMENT** — The suggestion would be pure taste or speculative architecture: "this should be CQRS" / "split the read and write models" / "add a separate read store to be ready" when the reads and writes are the same shape and scale and one model serves both correctly. Stay silent. **"This doesn't need CQRS" is the single most common correct outcome of this lens** — and recommending *against* an unnecessary separate read store, or *toward* a simple same-store projection, is itself a finding, not silence.

**Overriding rule:** every finding must name a concrete consequence — what breaks, what serves wrong data, what goes stale and is read as truth, what drifts undetected, what fails to scale, or what concretely improves. "This should be CQRS" / "the read and write models should be separate" / "this isn't real CQRS" is **not** a finding. "The order-confirmation screen reads the separate, asynchronously-updated read store immediately after the place-order command, so the customer who just placed an `Order` sees an empty list and places it again — a read-your-own-writes anomaly indistinguishable from a lost write" **is**. If no concrete consequence can be articulated, the finding is NO_COMMENT, no matter how far the design diverges from a textbook CQRS shape. Style (formatting, naming-case, import order, `this.`, whitespace) belongs to linters, never to review.

## Review Questions

Before raising or accepting a CQRS separation (or its absence), the reviewer should be able to answer:

- Do the read and write needs here genuinely diverge in shape, scale, or query complexity, or is this symmetric CRUD where one model serving both is honest and sufficient? If they don't diverge, why is there a separation at all?
- If there is a separation, is it the **lightweight** form (separate read/write models or methods against one store, no eventual consistency) or the **heavyweight** form (a separate read store updated asynchronously)? Is the heavier form actually warranted, or would a same-store read-only projection do?
- Is this CQRS being **conflated with Event Sourcing**? Is event-sourcing infrastructure being adopted because CQRS was wanted, when the two are independent and ES brings its own large cost?
- What is the read model **derived from**, and **who owns** building and maintaining the projection? Is there a **rebuild path** — can the read model be reconstructed from the source of truth when it drifts or its shape changes?
- If the read store is separate, **how is it kept in sync** — domain events, a transactional outbox, CDC — and is that mechanism reliable (no dual write that a crash can split)?
- The read side is **eventually consistent** with the write side: what is the staleness window, is it bounded and monitored, and where does a stale read cause a wrong decision?
- Is **read-your-own-writes** handled where it matters? After a user issues a command, does the next screen read a store that may not reflect it yet, and if so what countermeasure (read from the write side for that view, return the result in the command response, optimistic UI) prevents the change from appearing lost?
- Does the **command side ever read through the read model** to enforce an invariant? If so, is it deciding against a stale view, and can two concurrent commands both pass a check the fresh state would fail?
- Is a complex **reporting or display query being forced through the write/aggregate model**, paying an N+1 or whole-aggregate-load cost that a read projection would avoid?
- If there is no separation, is the absence correct — does a single model serve both paths without a named problem — or is the write model being genuinely contorted to serve a query it is the wrong shape for?

## Heuristics

### Lightweight vs Separate-Store CQRS (the Spectrum)

**What to look for:** Where on the spectrum a proposed separation actually sits, and whether the design jumped to the heavy end without needing to. The lightweight end is separate read and write models (or just read-only query methods) against **one store** — the command side saves aggregates, the query side reads through projections or `@Query` methods, one source of truth, reads as fresh as writes. The heavy end is a **separate read store** kept in sync **asynchronously**, with all the eventual-consistency and sync machinery that implies. The signal worth acting on is a design reaching for a separate read store, a read replica, or an event-driven projection pipeline when the query it needs could be served by a read-only projection against the same store.

**Why it matters:** The two ends of the spectrum have radically different costs. A same-store read-only projection costs almost nothing — it is a query shaped for the consumer, and it reads-its-own-writes for free because there is one store. A separate read store costs eventual consistency (reads lag writes), a projection process to build and operate, a sync mechanism to keep reliable, and two models that can drift. Jumping to the heavy form when the light form would do means the design pays all of that — and inherits a read-your-own-writes anomaly and a drift hazard — to solve a query-shape problem a projection in the same store already solves with neither. The concrete consequence is a system that is harder to operate and reason about, and that can show users stale data, in exchange for nothing the simpler form didn't offer.

**When NOT to comment:** When the read workload genuinely must scale independently of the write workload, or the read store must be a different technology shaped for its queries, a separate read store is the right end of the spectrum — do not push it back to a same-store projection that cannot meet the scale or shape requirement. When the design is already at the lightweight end (read-only projections against one store) and that is sufficient, there is nothing to flag; do not invent a need for a separate store. The choice along the spectrum is frequently a defensible judgment with no named consequence either way — that is NO_COMMENT.

**Modern Java/Spring idiom:** Lightweight — Spring Data interface or class/DTO projections, dedicated read-only query repositories or `@Query` methods returning view DTOs, or a database view, all against the same store the write model uses. Heavyweight — a separate read store (a different schema, a read replica, or a different database) updated asynchronously from domain events, a transactional outbox, or CDC. *Verify which the project actually has before recommending either; never assume a separate store, a read replica, or a sync pipeline is present.*

**Key review questions:** Which end of the spectrum is this, and does the query it serves actually need the heavy form? Could a read-only projection against the same store serve this without a separate store and without eventual consistency? If a separate store is proposed, what specifically requires it — scale, shape, or technology — that a same-store projection cannot provide?

**Example review comment:** "SHOULD: the design adds a separate read store updated from `Order` events just to serve the order-summary screen, which queries the same data the write store already holds. That introduces eventual consistency and a projection pipeline to operate, and the summary doesn't need either — a read-only Spring Data projection (or a view) against the write store would serve the same shape, read-its-own-writes for free, and have nothing to keep in sync. I'd start there and reserve a separate store for a read workload that genuinely must scale or be shaped independently. (Verify the project doesn't already have such a store.)"

### CQRS Is Not Event Sourcing

**What to look for:** Event-sourcing infrastructure or ceremony being introduced *because* CQRS was wanted — an event store, an append-only event log as the source of truth, aggregates rebuilt by replaying events, an ES/CQRS framework — when the actual driver was only "we want to read and write through different models." The two are constantly conflated; the tell is a design that justifies adopting Event Sourcing by appealing to CQRS, or a separate-store CQRS that assumes it *must* be event-sourced.

**Why it matters:** **CQRS and Event Sourcing are independent.** CQRS is about separating the read model from the write model; Event Sourcing is about persisting state as a sequence of events rather than as current state. You can do either without the other: CQRS with a perfectly ordinary state-stored write model (and a projection over it), and Event Sourcing without ever splitting the read and write models. Adopting Event Sourcing because you wanted CQRS is a real, costly mistake — Event Sourcing brings its own large burden (event versioning and schema evolution, snapshotting for replay performance, an event store to operate, the mental shift to event-as-truth, and harder ad-hoc querying) that CQRS does not require and that nothing in a read/write split asks for. The concrete consequence is a system carrying the full weight of event-sourcing infrastructure to get a read-model separation a plain projection over the existing state store would have delivered.

**When NOT to comment:** When the system has an *independent* driver for Event Sourcing — a genuine need for a full audit trail of every state change, temporal queries, or replay to rebuild state — then ES is justified on its own merits, and CQRS often pairs naturally with it (the event stream is a convenient thing to project read models from). Do not flag ES that is adopted for its own reasons. When CQRS is present with a plain state-stored write model and no ES in sight, that is the common, correct shape — there is nothing to comment on.

**Modern Java/Spring idiom:** Lightweight and heavyweight CQRS both work fine over a conventional JPA/JDBC state-stored write model — the read side is a projection or a separately-synced store, with no event store required. If the design proposes an ES/CQRS framework (for example Axon) or an event store, that is a large dependency and an architectural commitment; *recommend it only against a named Event-Sourcing driver, and verify it is actually present rather than assuming the project is event-sourced.*

**Key review questions:** Is Event Sourcing being adopted here for its own reasons (audit, temporal queries, replay), or only as a perceived prerequisite for CQRS? Does the write model need to be event-sourced at all, or would a plain state-stored model plus a projection deliver the read/write separation the design actually wants? If an ES/CQRS framework is proposed, what concrete driver justifies that dependency?

**Example review comment:** "MUST (architectural): the design introduces an event store and rebuilds the `Order` aggregate by replaying events, justified as 'needed for CQRS.' CQRS and Event Sourcing are independent — the read/write separation here only needs a read model projected from the write store, not the write model re-expressed as an event log. Event Sourcing brings event versioning, snapshotting, and an event store to operate, none of which the read/query split requires. Unless there's a real audit-trail or replay requirement driving Event Sourcing on its own, I'd keep a conventional state-stored write model and project the read model from it."

### Read-Model Projections and Ownership

**What to look for:** Who builds and owns the read-model projection, and how it is rebuilt. The trigger is any derived read model — a denormalized view, a precomputed summary, a separate read store — and specifically whether there is a clear owner of the projection logic and a defined path to **reconstruct the read model from the source of truth**. The strongest signal of a problem is a projection that exists with no named owner and no rebuild path: it was populated once and is now assumed correct forever.

**Why it matters:** A read model is **derived**, not authoritative — the write model (or the event stream) is the source of truth, and the read model is a consequence of it. That means two things must be true. First, someone must **own** the projection: the logic that turns write-side changes into read-side state, including which changes it reacts to and what it computes. An ownerless projection drifts as the write model evolves and no one updates the derivation. Second, the read model must be **rebuildable** from the source of truth — because it will drift (a missed event, a bug in the projection, a schema change to the read shape), and the only clean recovery is to rebuild it from the authoritative data. A projection with no rebuild path is a liability: when it drifts, the system serves wrong data and there is no way to correct it short of manual surgery, and the drift is often invisible until a user notices the read disagrees with reality. The concrete consequence is wrong data served confidently and a drift that is undiagnosable and unrecoverable.

**When NOT to comment:** A same-store read-only projection (a Spring Data projection, a plain non-materialized view) has nothing to rebuild — it is computed from the live source of truth on every query, so ownership and rebuild are non-issues; do not demand a rebuild path for a plain view. When a separate read store already has an owned, tested rebuild path (replay from events, re-derive from the write store), the requirement is met — do not pile on more. The rebuild/ownership concern applies specifically to a *materialized* read model — a separate read store, or a same-store materialized view — that can drift or go stale between refreshes.

**Modern Java/Spring idiom:** For a materialized read store, keep the projection logic in one owned place (a projector component that consumes domain events or reads the write store and writes the read model), and provide an explicit, tested **rebuild** operation that re-derives the read model from the source of truth — replaying events, or re-reading the write store — that can be run on demand. A database **materialized view** with a defined refresh is a simpler owned-and-rebuildable form where it fits. For a non-materialized projection (an interface/DTO projection or a plain view), the source of truth *is* the read, so no rebuild applies. *Verify what the project actually materializes before recommending rebuild machinery.*

**Key review questions:** What is this read model derived from, and who owns the logic that derives it? When (not if) it drifts from the write model, how is it rebuilt — is there a tested path to re-derive it from the source of truth? Is this a materialized projection that can drift, or a live view that is recomputed each query and cannot?

**Example review comment:** "MUST: the `Order` summary read store is populated by a projector on `Order` events, but there's no rebuild path — if the projector misses an event or a bug skews a computed total, the read store drifts from the write model and serves wrong order totals with no way to correct it short of hand-editing rows. A read model is derived data; it needs an owned, tested rebuild that re-derives it from the write store (or the event stream). Add a rebuild operation and confirm a single owner maintains the projection logic. (If this were a live view recomputed per query, this wouldn't apply — but it's a materialized store.)"

### Synchronizing the Read Model

**What to look for:** How a separate (materialized) read model is kept in sync with the write model, and whether that synchronization is reliable. The trigger is a separate read store and a mechanism that updates it: domain events, a transactional outbox, or CDC. The strongest signal of a problem is the read store updated by a **dual write** — the command method writes the write store and then writes (or publishes to) the read store in the same code path with no atomic guarantee between them.

**Why it matters:** Updating a separate read store is the same reliable-propagation problem the [`./saga.md`](./saga.md) lens treats for cross-boundary work: the write store and the read store (or the broker feeding it) are separate systems, so a single local transaction cannot span both. If the command saves the write model and then updates the read store as two separate operations, a crash between them leaves the read model **permanently inconsistent** with the write model — the write committed, the read update did not, and nothing reconciles them. The reliable mechanisms are the ones that lens describes: a **transactional outbox** (write the change as an outbox row in the same local transaction as the write-model change, then relay it asynchronously to the projector — durable exactly when the write committed), **domain events** consumed by a projector (with the same after-commit caveats), or **CDC** tailing the write store's log or the outbox to drive the projection. All of these make the propagation reliable; a bare dual write does not. The concrete consequence of getting this wrong is silent, permanent divergence between what the system decided and what it shows. See [`./saga.md`](./saga.md) for the outbox and at-least-once mechanics, and the [`cdc.md`](./cdc.md) lens for change-data-capture as a projection source.

**When NOT to comment:** When the read model is a same-store projection or a live view, there is nothing to synchronize — it is computed from the source of truth and cannot diverge; do not impose outbox or CDC machinery on a view. When the design already propagates reliably (an outbox the projector consumes, CDC over the write store) and consumers are idempotent, the requirement is met — do not demand a second mechanism. The synchronization concern is specific to a *separately-materialized* read store fed across a process or store boundary.

**Modern Java/Spring idiom:** Prefer a transactional outbox written in the same `@Transactional` method as the write-model change, relayed asynchronously to a projector that updates the read store; or CDC tailing the write store / outbox. In-process, `@TransactionalEventListener(AFTER_COMMIT)` can drive a projector, but note that listener runs **after the originating transaction has committed with no active transaction**, so a write to the read store inside it is non-transactional by default and may be silently lost unless the listener opens its own transaction (e.g. `@Transactional(propagation = REQUIRES_NEW)`) — and even then it is a second write that can fail independently, which is exactly why an outbox or CDC is more reliable for a separate store. Make the projector idempotent for the at-least-once redelivery these mechanisms imply. *Verify whether the project has an outbox, a broker, or a CDC pipeline before proposing one; never assume any are present.*

**Key review questions:** How does a write-model change reach the read store — an outbox, events, CDC, or a bare dual write? If the process crashes between the write-store commit and the read-store update, what reconciles them? Is the propagation durable exactly when the write commits, and is the projector idempotent under redelivery?

**Example review comment:** "MUST: `placeOrder` saves the `Order` to the write store and then, in the same method, writes the denormalized row to the read store — two writes with no shared transaction. A crash in between leaves the read store missing an order the write store has, permanently, with nothing to reconcile it. Write the change to an outbox row in the same transaction as the `Order`, and relay it asynchronously to the read-store projector (or use CDC over the write store) — the read update is then durable exactly when the order is. Keep the projector idempotent for the redelivery. The outbox mechanics are in the saga lens."

### Eventual Consistency and Read-Your-Own-Writes

**What to look for:** Whether the design accounts for the read side lagging the write side, and specifically for the canonical anomaly: a user issues a command and then immediately reads a view backed by an asynchronously-updated read store that does not reflect their change yet. The trigger is any flow where a command is followed by a read of a separately-synced read model — a confirmation screen, a redirect-to-list after a create, a "your changes" view rendered right after a save.

**Why it matters:** A separate read store updated asynchronously is **eventually consistent** with the write store — there is a window, however small, where a write has committed but the read model has not caught up. Most reads tolerate this fine (a dashboard a few seconds stale is harmless). The dangerous case is **read-your-own-writes**: the user who just made a change expects to see it, and an immediate read of the lagging read store shows the old state — the change *appears lost*. This is indistinguishable to the user from an actual lost write: they placed the `Order` and it's not in their list, so they place it again, or they file a support ticket, or they make a wrong decision on stale data. It is the single most common way eventual consistency bites a CQRS read side, and it is a correctness/UX defect, not a theoretical concern. The countermeasures are concrete: for the post-command view, **read from the write side** (the command's own freshly-written state) rather than the lagging read model; **return the result in the command response** so the UI renders what it just did without a round-trip to the read store; use an **optimistic UI** that shows the intended state immediately and reconciles when the read model catches up; or **wait/poll** for the projection to catch up before showing the view (heavier, rarely worth it). The point is that read-your-own-writes must be *designed for*, not assumed away.

**When NOT to comment:** When the read model is a same-store projection or a live view, there is no lag — reads are as fresh as writes and read-your-own-writes is automatic; do not raise staleness for a same-store read. When the lagging read is genuinely not read immediately after a command (an overnight `Report`, an analytics dashboard nobody cross-checks against a fresh action), the staleness is harmless and demanding a countermeasure is ceremony. The anomaly matters specifically where a user reads back their own just-committed change through a lagging store.

**Modern Java/Spring idiom:** For the post-command view, query the write side (the same store the command wrote) or render from the value the command handler returns, rather than the asynchronously-updated read store. Where the UI can, show the intended state optimistically and let the read model reconcile. Bound and monitor the projection lag so the staleness window is known. *None of this applies if the read is a same-store projection — verify whether the read store is actually separate and asynchronous before designing around lag.*

**Key review questions:** After a command, does the next screen read a store that may not reflect it yet? If so, what stops the user's own change from appearing lost — reading the write side for that view, returning the result in the response, or an optimistic UI? Is the staleness window bounded and monitored, and is there any read where stale data drives a wrong decision?

**Example review comment:** "MUST: after `placeOrder` commits to the write store, the confirmation screen reads the order list from the asynchronously-updated read store, which hasn't been projected yet. The customer sees their just-placed `Order` missing — indistinguishable from a lost order — so they place it again or contact support. Read-your-own-writes needs designing for here: render the confirmation from the command's own result (or read the write side for this one view), and reserve the read store for views where a few seconds of lag is harmless. (If the read were a same-store projection this wouldn't arise — confirm the store is actually separate and async.)"

### Querying the Write/Aggregate Model for Reporting

**What to look for:** A reporting, dashboard, or display query being served by loading and traversing the **write/aggregate model** — loading whole aggregates to read a few fields, joining across several aggregates the write model references only by identity, or iterating aggregates and lazily loading children. The trigger is a query whose shape fights the aggregate's shape: the aggregate is modeled to enforce invariants on writes, and the query wants a flat, cross-aggregate, aggregated view the aggregate was never shaped to provide. This is the case that *legitimately* motivates a read projection.

**Why it matters:** An aggregate is the write-side consistency boundary (see [`./ddd.md`](./ddd.md)) — it is shaped to load, mutate, and save a coherent unit atomically, not to answer arbitrary read queries. Forcing a complex read through it has concrete costs: an **N+1 storm** (iterate `Order`s, lazily load each one's `LineItem`s and `Customer` to build a row), **over-fetching** (load the whole `Order` aggregate graph to display three columns), and **cross-aggregate joins the write model can't express** (the write model references other aggregates by id per DDD, so a report spanning `Order`, `Customer`, and `Invoice` has no efficient path and ends up doing application-side joins). Under load this takes the query — and sometimes the write path sharing the same tables — down. This is exactly the divergence CQRS exists for: the read needs a shape the write model is wrong for, so a **dedicated read-only projection** (shaped for the query, read directly, bypassing the aggregate) is the right answer — and frequently it can be a same-store projection, not a separate store. The concrete consequence of not separating here is a slow, fragile query coupled to the write shape; the consequence of *over*-correcting is jumping to a separate read store when a same-store projection sufficed (see the spectrum heuristic).

**When NOT to comment:** When the query is simple and the aggregate's shape already fits it (load one `Order` by id and show it), there is no divergence — reading through the write model is correct and a projection is ceremony. When a read-only projection already serves the reporting query well, the case is handled — do not push it further toward a separate store without a scale or shape driver. The finding exists when a complex read genuinely contorts the write model and pays for it, not whenever a read touches an aggregate.

**Modern Java/Spring idiom:** Serve the reporting/display query with a **read-only projection** that selects exactly the columns the view needs, bypassing the aggregate — a Spring Data interface or DTO projection, a `@Query` returning a view DTO, or a database view — typically against the same store. Reach for a separate read store only when the read workload must scale or be shaped independently of the write workload (the spectrum heuristic). Avoid building report rows by iterating aggregates and lazy-loading their children. *Verify the persistence stack and whether projections are already in use before recommending a specific form.*

**Key review questions:** Is this query loading whole aggregates (or lazily loading children in a loop) to read a few fields, or joining across aggregates the write model only references by id? Does the read need a shape the invariant-enforcing write model is the wrong shape for? Would a read-only projection against the same store serve it directly — and is a *separate* store actually needed, or is a same-store projection enough?

**Example review comment:** "SHOULD: the daily `Report` is built by loading every `Order` aggregate and lazily fetching each one's `LineItem`s and `Customer` to assemble a row — an N+1 that will degrade sharply as orders grow, and it loads full aggregate graphs to read four columns. This is the read/write divergence CQRS is for: the report wants a flat, cross-aggregate shape the write model isn't built for. A read-only projection selecting just those columns (a Spring Data DTO projection or a view) against the same store would serve it directly without touching the aggregate. No separate read store needed unless the reporting load must scale independently — keep it lightweight."

## Anti-Patterns

- **Full separate-store CQRS on simple symmetric CRUD** — *Diff/design:* a separate, asynchronously-updated read store (or an event-driven projection pipeline) introduced for data whose reads and writes are the same shape and the same scale — a screen that shows back the fields it just saved. *Harm:* the design pays eventual consistency, a projection to build and operate, a sync mechanism to keep reliable, and two models that can drift, all to serve a query a single model against one store answered for free and consistently; it even gains a read-your-own-writes anomaly the single model never had. *Fix:* use one model against one store; if the reads ever want a different shape, add a read-only projection in the *same* store before considering a separate one. Introduce a separate read store only against a real scale, shape, or technology driver.

- **Conflating CQRS with Event Sourcing** — *Diff/design:* event-sourcing infrastructure (an event store, aggregates rebuilt by replay, an ES/CQRS framework) adopted because CQRS was wanted, with no independent Event-Sourcing driver. *Harm:* the system carries the full weight of Event Sourcing — event versioning and schema evolution, snapshotting, an event store to operate, harder ad-hoc querying — to obtain a read/write separation a plain projection over the existing state store would have delivered. *Fix:* keep a conventional state-stored write model and project the read model from it; adopt Event Sourcing only for its own reasons (a genuine audit-trail, temporal-query, or replay requirement), at which point CQRS may pair with it naturally.

- **Ignoring eventual consistency on a read-your-own-writes path** — *Diff/design:* a command is followed immediately by a read of a separate, asynchronously-updated read store that hasn't caught up, and the UI treats the stale read as truth. *Harm:* the user's own just-made change appears missing — indistinguishable from a lost write — so they retry the command, file support tickets, or make a wrong decision on stale data. *Fix:* design read-your-own-writes explicitly — render the post-command view from the command's own result or from the write side, or use an optimistic UI — and reserve the lagging read store for views where staleness is harmless; bound and monitor the lag.

- **A read projection with no owner or rebuild path** — *Diff/design:* a materialized read model (a separate read store, a denormalized view) populated once, with no single owner of the projection logic and no tested way to re-derive it from the source of truth. *Harm:* as the write model evolves or the projector misses a change, the read model silently drifts and serves wrong data, and the drift is undiagnosable and unrecoverable short of manual surgery. *Fix:* give the projection one owner and an explicit, tested rebuild that re-derives the read model from the write model (or event stream); a same-store live view, which is recomputed each query and cannot drift, sidesteps the problem entirely where it fits.

- **Forcing complex multi-aggregate reporting through the write model** — *Diff/design:* a reporting or display query served by iterating aggregates and lazily loading their children, or by application-side joins across aggregates the write model references only by identity. *Harm:* an N+1 storm and whole-aggregate over-fetch that is slow and degrades sharply under load (and can drag down the write path sharing the tables), with the report coupled to the write shape so a write-model change breaks it. *Fix:* serve the query with a dedicated read-only projection shaped for it (a Spring Data DTO/interface projection, a `@Query` view DTO, or a database view), typically against the same store; move to a separate read store only if the read workload must scale independently.

## Modernization (Java/Spring idioms)

Treat these as the idiomatic modern expressions of CQRS concepts — but **inspect the target project first and never assume a separate store, a broker, a CDC pipeline, or a framework is present.** Establish the Java language level, the Spring Boot version, the persistence stack, whether there is a separate read store or read replica at all, and the messaging/streaming technology actually in use (if any) before recommending anything. A suggestion the project cannot compile, or that assumes a store, a broker, or an event-sourcing framework it does not have, is worse than no suggestion.

Frame every version- or dependency-specific claim as *verify against the actual project; never assume a separate store, a broker, or a framework is present.*

- **Command side = aggregates + repositories.** The write model loads, mutates, and saves aggregates through their repositories, enforcing invariants — exactly the DDD write model from [`./ddd.md`](./ddd.md). Keep one repository per aggregate root; the command side is where the rules live.
- **Query side = read-only projections (the lightweight default).** Serve reads through Spring Data **interface-based or class/DTO projections**, dedicated read-only query repositories or `@Query` methods returning view DTOs, or **database views**, against the **same store**. These bypass the aggregate, fetch exactly the columns the view needs, and read-their-own-writes because there is one source of truth. This is the right first reach for a read/write shape divergence. *Verify the persistence stack supports the projection form you suggest.*
- **Materialized views** are a heavier same-store option — a precomputed, refreshable view where the query is expensive enough to justify materializing it. Owned-and-rebuildable by construction via their refresh. *Verify the database supports them and that the refresh cadence fits the staleness tolerance.*
- **A separate read store (the heavyweight form)** — a different schema, a read replica, or a different database — updated **asynchronously** via domain events (`ApplicationEventPublisher` / `@TransactionalEventListener`), a **transactional outbox**, or **CDC**. This is where eventual consistency, read-your-own-writes, projection ownership/rebuild, and idempotent projectors all become mandatory concerns. Note that **`@TransactionalEventListener(phase = AFTER_COMMIT)` runs after the originating transaction has committed, with no active transaction**, so a write to the read store inside it is non-transactional by default and may be silently lost unless the listener opens its own transaction (e.g. `@Transactional(propagation = REQUIRES_NEW)`) — consistent with [`./ddd.md`](./ddd.md) and [`./saga.md`](./saga.md); for a separate store, an outbox or CDC is more reliable than an after-commit dual write. *Verify a separate store, a broker, and a CDC pipeline are actually present before proposing any of them; never assume the project is event-sourced.*
- **Do NOT introduce a separate read database, a read replica, or an event-sourcing/CQRS framework** (for example Axon) without a named driver. A separate store earns its keep only against a real read-scalability, query-shape, or technology requirement; an ES/CQRS framework is a large architectural commitment that needs an independent Event-Sourcing reason, not a CQRS one. *Verify what the project actually has, and never recommend adding a store, a broker, or a framework without naming the concrete benefit for this project.*

When a single model against one store does the job, say so plainly — CQRS is not the goal; the read and write paths served with the least machinery the divergence actually requires is. The lightweight read-only projection in the same store is the right answer far more often than a separate read store.

## Suggested Comment Style

Keep comments respectful, consequence-first, and severity-honest. Lead with what concretely breaks or goes stale or drifts (or what improves), not with a pattern label — name the failure mode so the author can weigh the trade-off rather than feel judged for not following a textbook. State severity honestly; a NIT should read like a NIT, not a blocker. Offer the alternative, and **acknowledge the trade-off** — the eventual consistency a separate read store introduces, the projection it adds to maintain, the read-your-own-writes anomaly it creates. When a single model would do, say so directly and recommend *against* CQRS — or toward a lightweight same-store projection — and treat that as a finding, not silence.

Example openers:

- "Do the reads and writes here actually diverge enough to separate them?" (recommends restraint)
- "This read store is updated asynchronously, so right after the command the user reads stale state..." (names the read-your-own-writes anomaly)
- "What is this read model derived from, and how is it rebuilt when it drifts?" (asks the ownership/rebuild question)
- "CQRS and Event Sourcing are independent — does the write model need to be event-sourced at all?" (separates the two)
- "A read-only projection in the same store would serve this without a separate store or eventual consistency..." (proposes the lightweight form)
- "NIT (not a blocker): ..." (flags taste-level, optional)
- "Reads and writes are the same shape and scale here — one model serves both correctly. No change needed." (explicitly endorses)

Example comments (neutral-domain):

- "I don't think this needs CQRS. The `Customer` screen reads back the same fields it just wrote, at the same scale — a single model against one store serves both, reads-its-own-writes for free, and has no projection to keep in sync. A separate read store would add eventual consistency and a sync pipeline for nothing. I'd keep one model unless a read workload appears that genuinely must scale or be shaped differently."
- "SHOULD: this adds a separate read store to serve the order-summary view, but the view queries data the write store already holds at the same scale. A read-only Spring Data projection (or a view) against the write store would serve the same shape without eventual consistency and with nothing to synchronize. Reserve a separate store for a read workload that must scale independently."
- "MUST: the confirmation screen reads the asynchronously-updated read store immediately after `placeOrder`, so the customer sees their just-placed `Order` missing and re-places it. Render the confirmation from the command's own result (or read the write side for this view) so their own change can't appear lost; the read store is fine for views where a few seconds of lag is harmless."
- "MUST: the `Order` summary read store has no rebuild path — if the projector misses an event it drifts from the write model and serves wrong totals with no way to correct it. A materialized read model needs an owned, tested rebuild that re-derives it from the write store. (A same-store live view wouldn't have this problem — it's recomputed each query.)"
- "MUST (architectural): the event store and replay-rebuilt aggregates are justified as 'needed for CQRS,' but the two are independent — the read/write split here only needs a projection over the write store, not the write model re-expressed as events. Unless there's a real audit/replay driver for Event Sourcing on its own, a conventional write model plus a projection is far less to carry."

## Integration (architecture-review)

The consuming skill uses this lens as a judgment aid at system/design altitude, not a checklist to enforce:

- **Never comment because CQRS is absent.** A design that serves reads and writes through one model against one store is not a finding by itself — for symmetric CRUD it is the correct design. Absence is worth raising only when you can name the concrete consequence (a complex read genuinely contorting the write model and paying an N+1 cost a projection would avoid). And recommending *against* unnecessary CQRS — "one model is simpler and consistent here" — or *toward* a lightweight same-store read projection instead of a separate store, is a legitimate, frequent finding, not silence.
- **Comment only when there is a real, articulable consequence** — a stale read taken as truth on a read-your-own-writes path, a projection that drifts with no rebuild, a dual write that splits the read and write models, a separate store bought where a same-store projection sufficed, a reporting query taken down by an N+1 through the aggregate.
- **Never block on preference.** "This should be CQRS" / "split the read and write models" / "add a read store to be ready" without a named consequence is NIT at most, and usually NO_COMMENT. Speculative separation with no current driver is NO_COMMENT.
- **Always tag findings** `MUST` / `SHOULD` / `NIT` per [`../rules/severity-rubric.md`](../rules/severity-rubric.md) so the author can triage by severity. (NO_COMMENT is the silent fourth outcome and, for this lens, the most common one — frequently in the form "this doesn't need CQRS.")
- **A few strong findings beat many weak ones.** One well-justified MUST about a read-your-own-writes anomaly, a drifting projection, or a dual write between the two models is worth more than a list of "consider a read model" observations — drop the weak ones to NO_COMMENT rather than padding the review.
- This lens reasons about whether the read and write models should separate, what the read model derives from, how it stays in sync, and the staleness that introduces — *system/design altitude*. Aggregate-internal modeling and where the consistency boundary falls (the write model *is* the aggregates) belong to [`./ddd.md`](./ddd.md); the messaging/consistency mechanism that propagates changes to a read model — outbox, events, at-least-once, idempotency — belongs to [`./saga.md`](./saga.md) (and change-data-capture to the [`cdc.md`](./cdc.md) lens); local readability and small refactors belong to [`./clean-code.md`](./clean-code.md); pattern-level structural calls belong to [`./design-patterns.md`](./design-patterns.md); test quality belongs to [`./testing.md`](./testing.md). Defer to those rather than re-litigating their concerns here.
